# U11 — Correct issue-type classification

**Lane:** System · **Owner:** Yanna + Hector (joint) · **Evidence type:** Judge

## What it checks

Every issue's declared Jira type (Story, Task, Bug, Sub-task) matches what the work actually is. Evaluated across Stories, Tasks, Bugs, and standalone Sub-tasks.

## Why it matters

Forecast accuracy depends on consistent sizing. Consistent sizing depends on consistent classification. When Stories are tracked as Tasks, when scope-creep is labeled as Bugs, or when Sub-tasks run standalone, the throughput data becomes meaningless and the forecast runs on bad inputs.

## How the grader evaluates it

Judge-only. Claude reads issue title, description, and AC, and classifies against Nolte's definitions:

- **Story:** user or business-facing capability.
- **Task:** operational or technical work with concrete observable output (rare by policy).
- **Bug:** verified defect against previously agreed behavior (not new scope relabeled).
- **Sub-task:** execution-level work supporting a Story (never standalone).
