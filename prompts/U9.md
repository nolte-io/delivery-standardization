# U9 — Story defect classification

**Version:** 0.1
**Type:** Judge (J)
**Lane:** Upstream
**Owner:** Yanna

## Purpose

Determine whether a Story Defect maps to an agreed scenario in the parent Story, or whether it represents scope evolution (a new requirement surfacing late). Both are valid; the distinction is a spec-quality signal.

## Inputs

- `{{story_defect_key}}`
- `{{story_defect_description}}`
- `{{parent_story_key}}`
- `{{parent_story_scenarios_text}}` — parent Story's `## Scenarios` section at commitment time

## Evidence codes

| Code | Verdict |
|---|---|
| `DEFECT_MAPS_TO_SCENARIO` | PASS |
| `DEFECT_IS_SCOPE_EVOLUTION` | PASS (flagged separately in rollup) |
| `DEFECT_DESCRIPTION_INSUFFICIENT` | INSUFFICIENT_EVIDENCE |

## User prompt

```
Dimension: U9 — Story defect classification.

You classify whether this Story Defect maps to an agreed scenario in the parent Story, or whether it represents scope evolution (a new requirement not in the original contract). Both are valid outcomes. Neither fails the defect. The distinction feeds a separate rollup signal about specification quality.

A defect MAPS_TO_SCENARIO when its described failure can be matched to an existing Given/When/Then in the parent Story's scenarios at commitment time.

A defect is SCOPE_EVOLUTION when the behavior it describes was not specified in the parent Story's scenarios, even if the fix is desirable or necessary.

Do not judge whether the fix should be done. Judge only whether it was in-contract or out-of-contract.

Story Defect: {{story_defect_key}}
Defect description:
---
{{story_defect_description}}
---

Parent Story: {{parent_story_key}}
Parent Story scenarios at commitment:
---
{{parent_story_scenarios_text}}
---

Return JSON with verdict PASS and evidence_code set to DEFECT_MAPS_TO_SCENARIO or DEFECT_IS_SCOPE_EVOLUTION. Use INSUFFICIENT_EVIDENCE only if the defect description does not describe a concrete behavior.
```
