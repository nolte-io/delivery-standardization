# Y2 — Observable business difference describable

**Version:** 0.1
**Type:** Judge (J)
**Lane:** Upstream
**Owner:** Yanna

## Purpose

Confirm that how we will know this worked is defined before the work ships, not after.

## Inputs

- `{{story_key}}`
- `{{observable_impact_text}}` — text of the Story's `## Observable Impact` section

## Evidence codes

| Code | Verdict |
|---|---|
| `IMPACT_HAS_METRIC_THRESHOLD_AND_OBSERVATION` | PASS |
| `IMPACT_MISSING_METRIC` | FAIL |
| `IMPACT_MISSING_THRESHOLD` | FAIL |
| `IMPACT_MISSING_OBSERVATION_POINT` | FAIL |
| `IMPACT_VAGUE_OR_ASPIRATIONAL` | FAIL |
| `IMPACT_SECTION_EMPTY` | FAIL |

## User prompt

```
Dimension: Y2 — Observable business difference describable.

A passing impact statement contains three elements: (1) a named metric, (2) a threshold or directional target, and (3) either a measurement location (dashboard, query, system) or an observability mechanism (logs, event, API response). Example of passing: "P50 claim intake time drops from 8min to <3min, measured in Datadog dashboard X, observed within 2 weeks of release."

A failing statement is missing any of those three elements, uses aspirational language ("users will be happier," "the product will feel better"), or describes only the delivered artifact rather than the observable change it produces.

Story: {{story_key}}
Observable Impact text:
---
{{observable_impact_text}}
---

Grade Y2. Return JSON only.
```
