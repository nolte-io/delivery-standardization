# U11 — Issue-type classification

**Version:** 0.1
**Type:** Judge (J)
**Lane:** System
**Owner:** Yanna + Hector

## Purpose

Confirm the Jira Issue Type matches the nature of the work. Feeds the classification-correction signal in the rollup. Runs against Stories, Tasks, Bugs, and standalone Sub-tasks.

## Inputs

- `{{issue_key}}`
- `{{declared_type}}` — current Jira Issue Type
- `{{issue_title}}`
- `{{issue_description}}`
- `{{acceptance_criteria_if_any}}`

## Evidence codes

| Code | Verdict |
|---|---|
| `TYPE_CORRECT` | PASS |
| `TYPE_MISMATCH` | FAIL (`recommended_type` is populated) |

## Classification rules

- **Story** — user or business-facing capability; new behavior an end user or stakeholder can observe.
- **Task** — operational or technical work with a concrete observable output (infrastructure, upgrades, integrations, configuration). Rare by policy.
- **Bug** — verified defect where system behavior differs from previously agreed acceptance criteria. Not new scope relabeled.
- **Sub-task** — execution-level work supporting a Story. Never standalone delivery.

## User prompt

```
Dimension: U11 — Issue-type classification.

Read the issue and classify it against Nolte's type definitions:

- Story: user or business-facing capability; new behavior an end user or stakeholder can observe.
- Task: operational or technical work with a concrete observable output (infrastructure, upgrades, integrations, configuration). Must be operational, not value-delivery work in disguise.
- Bug: verified defect where system behavior differs from previously agreed acceptance criteria. Not new scope relabeled.
- Sub-task: execution-level work that supports a Story. Never standalone.

Common misclassifications to detect:
- A Task that describes user-facing behavior change → should be Story.
- A Bug that introduces new behavior rather than restoring agreed behavior → should be Story.
- A Story that describes a fix to previously-Done behavior → should be Bug.
- A Sub-task appearing without a parent Story while delivering value → should be Story.

Declared type: {{declared_type}}

Title:
{{issue_title}}

Description:

{{issue_description}}
---

{{acceptance_criteria_if_any}}

If declared_type matches the correct type, return PASS with evidence_code TYPE_CORRECT and recommended_type equal to declared_type.

Return JSON only.
```
